# Implementation Plan: Dynamic QR Platform for Agencies and Enterprises

**Branch**: `002-dynamic-qr-google-integration` | **Date**: 2026-03-23 | **Spec**: `specs/002-dynamic-qr-google-integration/spec.md`
**Input**: Feature specification from `specs/002-dynamic-qr-google-integration/spec.md`

## Summary

Deliver a scalable Dynamic QR backend with RESTful FastAPI services, strict soft
delete lifecycle for core entities, OAuth-enabled Google integrations, and
high-throughput scan handling. The redirect endpoint (`GET /q/{short_code}`) returns
HTTP 302 quickly while scan logging is handled asynchronously. Analytics dashboards
query pre-aggregated `daily_analytics_summary` data generated by scheduled jobs.
Google Calendar integrations also support month/year event retrieval, selective
campaign import, and campaign sync-state reconciliation for dashboard icons.

## Technical Context

**Language/Version**: Python 3.12  
**Primary Dependencies**: FastAPI, Pydantic v2, SQLAlchemy 2.x, Alembic, async MySQL driver, PyJWT, bcrypt, OAuth client libraries, Redis client, optional async queue client  
**Storage**: MySQL (primary), Redis (short-code cache), queue backend for async scan writes (implementation choice finalized in Phase 1)  
**Testing**: pytest, pytest-asyncio, API integration tests, contract tests, load tests for redirect path  
**Target Platform**: Linux-based API deployment behind load balancer  
**Project Type**: Backend web-service  
**Performance Goals**: `/q/{short_code}` p95 < 200 ms, sustain up to 1,000,000 scans/day, dashboard query p95 < 2 s from summary tables  
**Constraints**: RESTful API, soft deletes for core tables, JWT+bcrypt auth, RBAC (`admin`,`agency`,`user`) with tenant ownership boundaries, OAuth 2.0 secure token storage with encryption/rotation/auditing, materialized summary analytics, durable async scan logging with retries/DLQ  
**Scale/Scope**: Multi-tenant agency/enterprise campaigns with Google Analytics and Google Calendar integrations

## System Architecture & High-Level Design

### Request Flow

1. Client request reaches FastAPI router.
2. Router validates payloads with request schemas.
3. Service layer enforces business rules and RBAC boundaries (`admin` all-tenant,
   `agency` company-scoped, `user` creator-scoped).
4. Repository layer executes SQLAlchemy-based persistence operations.
5. Responses return with standardized REST status and response models.

### Redirect Logic (`GET /q/{short_code}`)

1. Resolve `short_code` to QR record (`destination_url`, status, UTM fields,
   `ga_measurement_id`) using Redis cache first, then MySQL fallback.
2. Build final redirect URL (destination + UTM params when configured).
3. Capture scan metadata (`ip_address`, `user_agent`, device/os/browser,
   country/city, referer) and enqueue to a durable async logging pipeline.
4. Return HTTP 302 immediately after durable enqueue; eventual persistence is
   guaranteed by retries and dead-letter handling.

### Scalability Strategy

- Redis caching for hot short-code lookups.
- Durable asynchronous scan log insertion via queue-backed pipeline to protect
  redirect latency while preserving delivery guarantees.
- Composite index strategy for scan write/query paths.
- Cron-based aggregation every 5 minutes (near-real-time) and end-of-day reconciliation
  into `daily_analytics_summary`.

## Database Schema (MySQL + SQLAlchemy)

The implementation MUST use exactly these 7 tables and constraints:

1) `users`
- Columns: `id`, `email` (UNIQUE), `password_hash`, `company_name`, `role`,
  `subscription_plan`, `created_at`, `updated_at`, `deleted_at`
- Constraints:
  - `email` unique
  - `role` CHECK in (`admin`, `agency`, `user`)

2) `user_integrations`
- Columns: `id`, `user_id` (FK), `provider_name`, `access_token`, `refresh_token`,
  `expires_at`
- Constraints:
  - `provider_name` CHECK in (`google_calendar`, `google_analytics`)
  - UNIQUE KEY (`user_id`, `provider_name`)
  - FK `user_id -> users.id`

3) `campaigns`
- Columns: `id`, `user_id` (FK), `name`, `description`, `start_date`, `end_date`,
  `status`, `created_at`, `updated_at`, `deleted_at`
- Constraints:
  - FK `user_id -> users.id` with ON DELETE CASCADE
  - Application policy: soft-delete in normal workflows; cascade applies only to
    controlled maintenance hard-delete operations

4) `qr_codes`
- Columns: `id`, `user_id` (FK), `campaign_id` (FK), `name`, `short_code` (UNIQUE),
  `destination_url`, `qr_type`, `design_config` (JSON), `ga_measurement_id`,
  `utm_source`, `utm_medium`, `utm_campaign`, `status`, `created_at`, `updated_at`,
  `deleted_at`
- Constraints:
  - `short_code` unique
  - FK `campaign_id -> campaigns.id` with ON DELETE SET NULL
  - FK `user_id -> users.id`

5) `qr_event_details`
- Columns: `id`, `qr_id` (FK UNIQUE 1-to-1), `event_title`, `start_datetime`,
  `end_datetime`, `location`, `description`, `google_event_id`, `updated_at`
- Constraints:
  - UNIQUE KEY on `qr_id`
  - FK `qr_id -> qr_codes.id`

6) `scan_logs`
- Columns: `id` (BIGINT), `qr_id` (FK), `scanned_at`, `ip_address`, `user_agent`,
  `device_type`, `os`, `browser`, `country`, `city`, `referer`
- Constraints and indexes:
  - FK `qr_id -> qr_codes.id`
  - INDEX (`qr_id`, `scanned_at`) required

7) `daily_analytics_summary`
- Columns: `id`, `qr_id` (FK), `summary_date`, `total_scans`, `unique_visitors`
- Constraints:
  - UNIQUE KEY (`qr_id`, `summary_date`) for UPSERT
  - FK `qr_id -> qr_codes.id`

## High-Traffic Architecture Details

### Redirect Logic (`/q/{short_code}`)

- Resolve `destination_url` and UTM context (`utm_source`, `utm_medium`,
  `utm_campaign`, optional `ga_measurement_id`).
- Build redirect target.
- Dispatch scan-log write to a durable asynchronous queue.
- Return HTTP 302 immediately after durable enqueue.
- Guarantee eventual persistence through retry policy and dead-letter queue.

### Analytics Cronjob

- Scheduled task runs every 5 minutes for near-real-time updates and daily
  reconciliation run for correctness.
- Reads incremental `scan_logs` windows.
- Aggregates by (`qr_id`, date).
- UPSERTs into `daily_analytics_summary` using unique key (`qr_id`, `summary_date`).
- Job is idempotent and safe for reruns/backfills.

## Calendar Sync and Reconciliation Design

### Scope

- Pull events from Google Calendar by month/year window.
- Allow partial/multi-select import into campaigns.
- Persist campaign-level calendar link/sync metadata.
- Surface sync icon state in dashboard campaign list responses.

### Data Strategy

- Keep the 7-table architecture and extend `campaigns` with calendar-link metadata:
  - `google_event_id` (nullable, indexed)
  - `calendar_sync_status` (`not_linked`, `synced`, `out_of_sync`, `removed`)
  - `calendar_last_synced_at` (nullable datetime)
  - `calendar_sync_hash` (nullable string for drift detection)
- Enforce owner-scoped uniqueness on linked event ids with app-level checks.
- Use deterministic hash of mapped fields (name/description/date range/status) to
  detect local-vs-remote drift.

### API Additions

- `GET /api/v1/integrations/google-calendar/events`
  - Query: `range_type=month|year`, `year`, optional `month`
  - Response includes event payload + link candidate status.
- `POST /api/v1/integrations/google-calendar/import-campaigns`
  - Body contains selected `event_ids` and optional import policy
  - Behavior: create or update campaigns idempotently.
- `POST /api/v1/campaigns/{campaign_id}/calendar/sync`
  - Push local campaign updates to Google Calendar and set `synced`.
- `DELETE /api/v1/campaigns/{campaign_id}/calendar/link`
  - Remove event from Google Calendar and set local state `removed`.

### Reconciliation Strategy

- On event-list fetch and sync actions, compare `calendar_sync_hash` with freshly
  computed remote hash to classify state.
- Dashboard campaign list includes icon-ready sync status.
- Reconciliation can run on-demand and optional background sweep schedule.

## API Contract Focus

- Auth: JWT login/register, role claims propagation.
- Integrations: OAuth connect/callback/refresh/revoke for Google providers.
- Integrations: OAuth lifecycle plus month/year event fetch and selective import.
- Campaigns/QR: REST CRUD with soft-delete semantics.
- Campaign Calendar: sync, unlink/remove, and status projection for dashboard icon.
- Redirect: `GET /q/{short_code}` optimized for latency.
- Analytics: dashboard endpoints query summary table first.

## Finalized RBAC and Maintenance Audit Policy Notes

- RBAC scope enforcement remains strict and role-bound:
  - `admin`: cross-tenant access.
  - `agency`: company-scoped access.
  - `user`: creator-scoped access.
- Standard delete operations for `users`, `campaigns`, and `qr_codes` are logical
  deletes via `deleted_at`; physical deletion remains a controlled maintenance path.
- Maintenance hard-delete operations must be explicitly audited with actor identity,
  resource type/id, and reason metadata.
- OAuth credential operations (access, refresh, revoke) must emit audit events for
  traceability and security review.
